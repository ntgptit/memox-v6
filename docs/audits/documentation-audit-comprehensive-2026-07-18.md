## A. Executive verdict

### **APPROVED WITH CHANGES — conditional. Foundation may start now; the learning UI cannot until Product Owner decisions land.**

The documentation set is, in the main, genuinely strong: internally consistent business
contracts, an exemplary design-token layer, a coherent 8-box SRS policy that business and design
agree on exactly, idiomatic Riverpod/Clean-Architecture ADRs, and an acyclic WBS with an explicit
first-learning critical path. Automated structure checks pass (341 files, 0 broken links, 0
duplicate WBS IDs; design checklist 48 groups/288 items; guard 0 errors). **But the audit found a
material, previously-unreported wall of Business↔Design divergences — the exact class of conflict
`AGENTS.md`'s STOP-and-ask rule exists to catch — concentrated in the content-creation and study
screens.** These are not style nits; several are data-loss-grade (cascade delete, last-write-wins
sync) and several silently drop business-owned capabilities.

| Question | Answer |
|---|---|
| **Can implementation start?** | **Partially.** Foundation (tokens→theme→shell→persistence ports→DI→deterministic clock/ID) has no blocking conflict and can start immediately. |
| **Can foundation start?** | **Yes.** Sections 0–3 of the WBS are decision-complete enough to begin; fix E-002 (guard `viewmodels/` vs folder contract) and the pubspec dependency gap (D1) first. |
| **Can the learning flow start?** | **Blocked at the UI layer.** The domain/business contracts for the learning flow are implementable; the *screens* are blocked by P0 conflicts on modes, duplicates, language-pair delete, and first-launch routing. Domain + repository + SRS can be built in parallel while decisions are pending. |
| **What decision gates are blocking?** | 17 consolidated Business↔Design/Business↔Business decisions (see §E) + 2 design-kit integrity gates (scope self-contradiction; unverifiable "FIXED" evidence). |

**Bottom line:** the previous remediation report (`docs/audits/documentation-remediation-2026-07-18.md`)
declared nearly everything "Closed / APPROVED WITH CHANGES APPLIED." That verdict is **not
substantiated** for the design↔business reconciliation axis or the component-contract layer — this
audit surfaced 36 P0 and 28 P1 findings the prior pass did not report. Treat that report's "Closed"
column as covering the *domain-doc* remediation, not the *cross-corpus* reconciliation.

## C. 5-Why root-cause analyses

### 5Why-1 — Documentation source-of-truth
**Problem:** A prior audit says "Closed/approved"; this audit finds 64 open P0/P1s.
1. *Why weren't these caught?* The prior pass verified the business docs and the design kit **each in isolation**, plus automated link/ID structure — it never diffed business capability-by-capability against the design mocks.
2. *Why not?* There is no artifact that enumerates "for capability X: business owner ↔ design screen ↔ WBS item ↔ AC ↔ test." The traceability register (`work-item-register.md`) traces only the *first-learning chain*, not the full capability surface.
3. *Why is the chain partial?* The register was scoped to unblock foundation, not to be a complete matrix — a deliberate, reasonable MVP scope that was never widened.
4. *Why does that matter now?* Because the design kit was authored as a self-contained React/JSX prototype with its own "source of truth" language (`specs/*.md` say "auto-derived from source"), it drifted from business independently, and nothing forced reconciliation.
5. **Root cause / decision unlocked:** *No enforced Business↔Design reconciliation gate exists.* **Decision:** make a full capability traceability matrix (this report's §F is the seed) a required, versioned artifact, and add a "reconciled-with-business" checkbox to each design spec's front-matter before it is treated as buildable.
**Impact if ignored:** engineers build from whichever corpus they open first; data-loss defects (cascade delete, LWW sync) reach code.

### 5Why-2 — Business ↔ Design divergence
**Problem:** ~15 capabilities describe different behavior in business vs design.
1. *Why?* The design kit deliberately *simplified* several flows (audio→auto-generate only, translations→single, duplicates→2-way, rename→1 field) as prototype pragmatism.
2. *Why weren't the simplifications ratified?* They were made in JSX without a decision record; the business docs were never updated to match, nor were the simplifications rejected.
3. *Why no decision record?* The design kit's governance (`governance/`) tracks token/component promotion, not capability-scope reductions against business.
4. *Why does the split persist?* `AGENTS.md`'s STOP-and-ask rule fires at *development* time, but no one has developed these screens yet — so the conflicts sat latent.
5. **Root cause / decision unlocked:** *Capability scope for v1 is genuinely undecided, and the design mock has been silently acting as the decision-maker.* **Decision:** Product Owner must rule each §E conflict as "business stands" (redesign) or "design stands" (narrow business) — do **not** let the mock be the decision.
**Impact if ignored:** silent capability loss (attach audio, merge duplicates, multi-translation) and provider pre-commitment (Google auth) ship without Product/Security sign-off.

### 5Why-3 — Ability to start the learning flow early
**Problem:** "start learning" is the #1 priority yet its screens are P0-blocked.
1. *Why?* The five graded modes' design specs collapse to a single flat `complete` state, omitting the business-canonical `round-complete`/`retry-round` mastery-loop states (and Guess's `invalid-distractor-pool`).
2. *Why?* The mastery-round loop is a behavioral invariant ("unlimited rounds until a clean round") that is easy to state in prose but was never storyboarded as screens.
3. *Why?* The design kit modeled each mode as one happy-path screen + a completion screen, matching a demo, not the retry mechanic.
4. *Why does it block?* Activation (Box 0→1) is gated on completing all modes' mastery loops; with no design for the loop, Flow 6→7 (the heart of "start learning") has no reference.
5. **Root cause / decision unlocked:** *The core learning mechanic (mastery-round retry) has no design surface.* **Decision:** either commission `round-complete`/`retry-round`/`invalid-distractor-pool` states for the four graded modes, or document that the loop is intentionally invisible and prove activation is testable without a per-round screen. **The domain/SRS/session logic is unblocked and should be built now in parallel.**
**Impact if ignored:** engineers ship single-pass modes; SRS activation criteria become untestable against the UI.

### 5Why-4 — Clean Architecture / Riverpod foundation
**Problem:** The guard's canonical paths and the WBS folder contract disagree; pubspec lacks the accepted stack.
1. *Why?* The guard config expects `lib/presentation/features/<f>/viewmodels/*.dart` (E-002) while WBS §4.2 forbids `viewmodels/` under a feature root (allows `providers/` only).
2. *Why?* The guard ruleset predates (or diverged from) the WBS's ratified layering; the guard also still targets a retired `folders/` path (D2-001) that two docs claim was already cleaned up.
3. *Why does pubspec lack Drift/go_router (D1)?* The ADRs "Accept" Drift+GoRouter but the scaffold pubspec was never updated, and the activation-debt plan tracks missing *source paths*, never missing *dependencies*.
4. *Why does that matter?* FD-08/FD-09 (DI + presentation state) for three critical-path features can't satisfy both the folder contract and the guard simultaneously.
5. **Root cause / decision unlocked:** *The guard config, the architecture contract, and the pubspec are three sources of truth for the same layering, not yet reconciled.* **Decision:** WBS item 0.4 ("Align guard upstream") must (a) pick `providers/` vs `viewmodels/` and make guard+WBS agree, (b) add Drift/go_router/companions to pubspec, (c) fix the stale `folders/` scope.
**Impact if ignored:** first feature PR fails the guard or the review; DI wiring stalls.

### 5Why-5 — Testability & guard compliance
**Problem:** Design "FIXED" evidence and several verification commands don't exist on disk.
1. *Why?* ~12 issue-register "FIXED" rows (incl. the sole P0, keyboard a11y) cite `tool/ui_kit_shots/*.mjs`, `tool/a11y/*.mjs`, `evidence/*.png` — none of which exist (only `tool/verify/run.mjs` is present).
2. *Why?* The design kit was authored in a JS/npm prototype repo with its own tooling; only the docs were vendored into this Flutter repo, not the tools.
3. *Why does it matter?* "0 open P0" depends on that keyboard-a11y fix being real and reproducible; it can't be inspected.
4. *Why not caught?* The register's own `ISS-REB-TOOL-01` admits the tooling is missing, but the FIXED rows that *depend* on that tooling were not downgraded.
5. **Root cause / decision unlocked:** *The design kit's evidence pipeline is aspirational, not present.* **Decision:** either port the shot/a11y tooling into this repo and regenerate, or downgrade the tool-dependent FIXED rows to OPEN and re-verify via runtime Flutter evidence once the app exists.
**Impact if ignored:** the design gate's "0 P0" is unverifiable; a11y regressions (the ariaLabel/ligature defects, the 44px vs 48px token) ship unmeasured.

### 5Why-6 — WBS dependency / critical path
**Problem:** The CP marking contradicts the WBS's own dependency graph.
1. *Why?* Five items (0.6, 1.7, 1.8, 2.9, 3.8) are prerequisites of CP items (incl. the terminal gate 5.7.5, which lists 1.8 directly) yet carry no CP marker.
2. *Why?* CP was applied by hand to the "obvious" spine; the range-dependencies (`1.1–1.7`, `2.2–2.9`) were not expanded when marking.
3. *Why does it matter?* Anyone staffing the critical path from the CP column under-scopes it and may deprioritize guard-profile/fixtures work that gates the release-gate DoD.
4. *Why not caught?* The automated check verifies ID *uniqueness*, not CP *closure under depends-on*.
5. **Root cause / decision unlocked:** *CP is hand-maintained and unverified against the graph.* **Decision:** either mark 0.6/1.7/1.8/2.9/3.8 CP or tighten the ranges, then add a CP-closure check to the verifier.
**Impact if ignored:** late discovery that the release gate can't close because a "non-CP" prerequisite slipped.
